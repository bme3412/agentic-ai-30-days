Waiting Strategies

How agents wait for pages to be ready - click each strategy to see the code

Network Idle ✓ Best

Wait until no network requests for 500ms. Most reliable for dynamic content.

await page.wait_for_load_state("networkidle")
Wait for Selector ✓ Precise

Wait for specific element to appear. Fast and precise when you know what to expect.

await page.wait_for_selector(".results-grid")
Wait for Function ✓ Custom

Wait for custom condition. Powerful for complex page state checks.

await page.wait_for_function("document.readyState === 'complete'")
Fixed Sleep ✗ Avoid

Fixed delays are fragile and slow. Don't use time.sleep() - wait for actual conditions!

time.sleep(2) # ❌ BAD: Guessing timing

DOM-Based vs Visual Grounding

When to use each approach

DOM-Based (Playwright)

  • Fast (< 100ms)
  • No API cost
  • Works with ARIA/accessibility
  • Reliable on standard HTML
  • Fails on canvas elements
  • Fails on PDFs/images
  • Requires accessible DOM

Visual Grounding (GPT-4o)

  • Works on ANY visual UI
  • Handles canvas/PDF/images
  • No DOM required
  • Natural language queries
  • Slow (~2-5s)
  • API costs ($0.01/call)
  • Less precise positioning

Error Recovery Strategies

How agents handle failures - click each card to see the strategy

⏱️ Transient Errors
Example: Network timeout, rate limit
Strategy: Retry with exponential backoff (wait 1s, 2s, 4s, 8s)
🔄 Recoverable Errors
Example: Element not found, stale reference
Strategy: Re-navigate to page and try again
🚫 Blockers
Example: CAPTCHA, login required, paywall
Strategy: Escalate to human intervention
⚠️ Logic Errors
Example: Invalid state, assertion failure
Strategy: Log error details and stop gracefully

Example 1: Playwright Basics 
from playwright.async_api import async_playwright

async with async_playwright() as p:
    browser = await p.chromium.launch()
    page = await browser.new_page()

    # Navigate to page
    await page.goto("https://example.com")

    # Wait for network to be idle
    await page.wait_for_load_state("networkidle")

    # Click a button
    await page.get_by_role("button", name="Submit").click()

    # Verify navigation
    assert page.url != "https://example.com"

    await browser.close()
Key Concepts:
  • Always wait for networkidle after navigation
  • Use ARIA roles for stable element selection
  • Verify state changes after actions
Example 2: ARIA-First Selection Hierarchy 
# Most stable → Least stable

# 1. ARIA role + name (best - survives redesigns)
submit = page.get_by_role("button", name="Add to cart")

# 2. Test ID (very stable if app has them)
checkout = page.get_by_test_id("checkout-btn")

# 3. Visible text (readable but fragile)
link = page.get_by_text("Sign in")

# 4. CSS selector (breaks on class renames)
element = page.locator("#submit-btn")

# 5. XPath (positional - avoid!)
xpath_elem = page.locator("//div[@class='container']/button[2]")

# Example: Prioritize ARIA, fallback to CSS
try:
    await page.get_by_role("button", name="Submit").click()
except:
    await page.locator(".submit-button").click()
Best Practices:
  • Always try ARIA first - it's semantic and stable
  • Test IDs are great if your team adds them
  • Avoid XPath unless absolutely necessary
  • Visual grounding is the escape hatch for everything else
Example 3: Multi-Step Workflow with State Tracking 
from playwright.async_api import async_playwright

async def search_github(query: str):
    state = {"step": "init", "results": []}

    async with async_playwright() as p:
        browser = await p.chromium.launch()
        page = await browser.new_page()

        # Step 1: Navigate
        state["step"] = "navigate"
        await page.goto("https://github.com/search")
        await page.wait_for_load_state("networkidle")

        # Step 2: Fill search
        state["step"] = "search"
        await page.fill('input[name="q"]', query)
        await page.get_by_role("button", name="Search").click()
        await page.wait_for_selector(".search-results")

        # Step 3: Extract results
        state["step"] = "extract"
        repos = await page.locator(".repo-list-item").all()
        for repo in repos[:5]:
            name = await repo.locator("a").first.text_content()
            state["results"].append(name)

        # Step 4: Verify
        state["step"] = "verify"
        assert len(state["results"]) > 0, "No results found"

        state["step"] = "complete"
        await browser.close()

    return state["results"]
State Tracking Benefits:
  • Know exactly where failures occur
  • Can resume from checkpoints
  • Easier debugging with step labels
  • Verification at each stage
Example 4: Visual Grounding Fallback 
from playwright.async_api import async_playwright, PlaywrightError
from openai import OpenAI

client = OpenAI()

async def click_with_fallback(page, aria_label: str):
    """Try DOM selection first, fall back to visual grounding."""

    # Attempt 1: DOM-based selection (fast, cheap)
    try:
        await page.get_by_role("button", name=aria_label).click()
        return {"method": "dom", "success": True}
    except PlaywrightError:
        pass

    # Attempt 2: Visual grounding (slow, costs API credits)
    screenshot = await page.screenshot()

    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{
            "role": "user",
            "content": [
                {"type": "text", "text": f"Find the coordinates of the '{aria_label}' button"},
                {"type": "image_url", "image_url": {"url": f"data:image/png;base64,{screenshot}"}}
            ]
        }]
    )

    # Parse coordinates from LLM response
    # Expected format: "The button is at (x=120, y=300)"
    coords_text = response.choices[0].message.content
    x = int(coords_text.split("x=")[1].split(",")[0])
    y = int(coords_text.split("y=")[1].split(")")[0])

    # Click at coordinates
    await page.mouse.click(x, y)

    return {"method": "visual", "success": True, "coords": (x, y)}
When Visual Grounding Shines:
  • Canvas-based UIs (Figma, Miro, graphics apps)
  • PDF viewers without accessible DOM
  • Shadow DOM or iframes blocking access
  • Legacy apps with poor accessibility
Example 5: browser-use Agent (High-Level) 
from browser_use import Agent
from langchain_openai import ChatOpenAI

# Create an agent with natural language task
agent = Agent(
    task="Find the top 3 Hacker News posts about AI agents today and extract their titles and URLs",
    llm=ChatOpenAI(model="gpt-4o")
)

# Run the agent - it handles all browser automation
result = await agent.run()

print("Agent completed task!")
print(f"Posts found: {result}")

# The agent autonomously:
# 1. Navigated to news.ycombinator.com
# 2. Searched for relevant posts
# 3. Extracted titles and URLs
# 4. Formatted the results
# 5. Returned structured data
browser-use Benefits:
  • No need to write Playwright code - just describe the task
  • Automatic visual grounding with GPT-4o
  • Self-correction and retry logic built-in
  • Best for one-off tasks and prototyping
